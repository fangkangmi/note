The problem

Reviewing a spec against APIE is manual toil

  • Open the PR. Open the APIE standard.
  • Scroll. Compare clause by clause. By hand.
  • Hope you didn't miss one.
  • Every reviewer does it slightly differently.

What if the standard checked itself?

Ask the room: "Honestly, how long does a proper APIE review of an OpenAPI PR take you?" Take 2-3 answers. Then: "Hands up if you're confident you catch every violation, every time." Few hands — that's the hook.

The real finding · pass@k validated

Run it with and without the skill

With the skill

Deep on APIE — cites clauses, catches standard violations a plain model can't know. But it narrows the model's view to the standard.

Without the skill

Broader, unfocused — catches general issues outside APIE that the skill's narrow lens skips over.

Both outputs are valuable. Use both lenses.

This is the honest, validated core message. We ran pass@k (k repeats): the skill reliably ADDS APIE depth AND reliably tunnels the model's attention to the standard, missing some general issues the no-skill run catches. That's exactly WHY we show both today: skill = depth on your standard, no-skill = breadth. Keep a human approver over both.
